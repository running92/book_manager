# 图书管理系统数据库表设计文档

## 1. 数据库选型说明

本系统采用 **SQLite** 作为数据库，原因如下：

1. SQLite 轻量级，部署简单，适合毕业设计项目快速搭建与演示。  
2. 不需要额外安装数据库服务，项目开箱即用。  
3. 对于图书管理系统这类中小型业务场景，SQLite 已能够满足基本的数据存储需求。  
4. 与 Flask 配合方便，易于本地开发、测试和答辩展示。  

本系统后端通过 **Flask + SQLAlchemy** 访问 SQLite 数据库，实现对用户、图书、分类、借阅记录等数据的统一管理。

---

## 2. 数据库总体设计

系统数据库主要包含以下 4 张核心业务表：

1. `users`：用户表  
2. `categories`：图书分类表  
3. `books`：图书信息表  
4. `borrow_records`：借阅记录表  

其中：

- `users` 表用于存储管理员和普通用户信息；
- `categories` 表用于存储图书分类；
- `books` 表用于存储图书基础信息及库存信息；
- `borrow_records` 表用于存储图书借阅与归还记录。

表之间的关系如下：

- 一个分类可以对应多本图书，属于 **一对多关系**；
- 一个用户可以产生多条借阅记录，属于 **一对多关系**；
- 一本图书可以对应多条借阅记录，属于 **一对多关系**。

## 3. 数据表详细设计

### 3.1 用户表（users）

#### 3.1.1 表说明

用户表用于存储系统登录用户的信息，包括管理员与普通读者两类角色。

#### 3.1.2 字段设计

| 字段名 | 类型 | 主键 | 非空 | 默认值 | 说明 |
|---|---|---:|---:|---|---|
| id | Integer | 是 | 是 | 自增 | 用户ID |
| username | String(50) | 否 | 是 | - | 登录用户名，唯一 |
| password_hash | String(255) | 否 | 是 | - | 加密后的密码 |
| real_name | String(100) | 否 | 否 | - | 真实姓名 |
| role | String(20) | 否 | 是 | reader | 用户角色：admin / reader |
| phone | String(20) | 否 | 否 | - | 手机号 |
| email | String(100) | 否 | 否 | - | 邮箱 |
| status | String(20) | 否 | 是 | active | 状态：active / disabled |
| created_at | DateTime | 否 | 是 | 当前时间 | 创建时间 |
| updated_at | DateTime | 否 | 是 | 当前时间 | 更新时间 |

#### 3.1.3 约束说明

- `id` 为主键，自增；
- `username` 唯一，防止重复注册；
- `password_hash` 存储加密后的密码，不允许明文保存；
- `role` 用于权限控制；
- `status` 用于标记用户是否可正常使用系统。

---

### 3.2 图书分类表（categories）

#### 3.2.1 表说明

分类表用于管理图书所属类别，支持中英文名称，便于实现双语展示。

#### 3.2.2 字段设计

| 字段名 | 类型 | 主键 | 非空 | 默认值 | 说明 |
|---|---|---:|---:|---|---|
| id | Integer | 是 | 是 | 自增 | 分类ID |
| name_zh | String(100) | 否 | 是 | - | 分类中文名称 |
| name_en | String(100) | 否 | 是 | - | 分类英文名称 |
| description | String(255) | 否 | 否 | - | 分类描述 |
| sort_order | Integer | 否 | 否 | 0 | 排序号 |
| created_at | DateTime | 否 | 是 | 当前时间 | 创建时间 |

#### 3.2.3 约束说明

- 分类名称应尽量唯一；
- 若分类下已有图书，则不允许直接删除该分类。

---

### 3.3 图书表（books）

#### 3.3.1 表说明

图书表用于存储图书基础信息、双语标题与简介、库存数量、馆藏位置等内容，是系统最核心的业务表之一。

#### 3.3.2 字段设计

| 字段名 | 类型 | 主键 | 非空 | 默认值 | 说明 |
|---|---|---:|---:|---|---|
| id | Integer | 是 | 是 | 自增 | 图书ID |
| isbn | String(50) | 否 | 否 | - | 国际标准书号 |
| title_zh | String(200) | 否 | 是 | - | 图书中文名称 |
| title_en | String(200) | 否 | 否 | - | 图书英文名称 |
| author | String(100) | 否 | 是 | - | 作者 |
| publisher | String(100) | 否 | 否 | - | 出版社 |
| publish_date | Date | 否 | 否 | - | 出版日期 |
| category_id | Integer | 否 | 是 | - | 分类ID，外键 |
| cover_image | String(255) | 否 | 否 | - | 封面图片路径 |
| description_zh | Text | 否 | 否 | - | 中文简介 |
| description_en | Text | 否 | 否 | - | 英文简介 |
| total_stock | Integer | 否 | 是 | 0 | 总库存 |
| available_stock | Integer | 否 | 是 | 0 | 当前可借库存 |
| location | String(100) | 否 | 否 | - | 馆藏位置 |
| status | String(20) | 否 | 是 | available | 状态：available / disabled |
| is_deleted | Integer | 否 | 是 | 0 | 逻辑删除标记 |
| created_at | DateTime | 否 | 是 | 当前时间 | 创建时间 |
| updated_at | DateTime | 否 | 是 | 当前时间 | 更新时间 |

#### 3.3.3 约束说明

- `category_id` 关联分类表；
- `total_stock >= available_stock >= 0`；
- 图书借出时 `available_stock` 减 1；
- 图书归还时 `available_stock` 加 1；
- 若图书已有借阅记录，建议采用逻辑删除，而不是直接物理删除。

---

### 3.4 借阅记录表（borrow_records）

#### 3.4.1 表说明

借阅记录表用于存储用户借书、归还、逾期等全过程信息，是系统实现借阅业务闭环的关键。

#### 3.4.2 字段设计

| 字段名 | 类型 | 主键 | 非空 | 默认值 | 说明 |
|---|---|---:|---:|---|---|
| id | Integer | 是 | 是 | 自增 | 借阅记录ID |
| user_id | Integer | 否 | 是 | - | 用户ID，外键 |
| book_id | Integer | 否 | 是 | - | 图书ID，外键 |
| borrow_date | DateTime | 否 | 是 | 当前时间 | 借阅时间 |
| due_date | DateTime | 否 | 是 | - | 应归还时间 |
| return_date | DateTime | 否 | 否 | - | 实际归还时间 |
| status | String(20) | 否 | 是 | borrowed | 状态：borrowed / returned / overdue |
| remark | String(255) | 否 | 否 | - | 备注 |
| created_at | DateTime | 否 | 是 | 当前时间 | 创建时间 |
| updated_at | DateTime | 否 | 是 | 当前时间 | 更新时间 |

#### 3.4.3 约束说明

- `user_id` 关联用户表；
- `book_id` 关联图书表；
- 借书时生成一条新记录；
- 归还时更新记录状态与归还时间；
- 若当前时间超过 `due_date` 且未归还，则状态应标记为 `overdue`。

## 4. 数据表关系设计

### 4.1 users 与 borrow_records

- 一个用户可以借阅多本图书；
- 一个用户对应多条借阅记录；
- 关系类型：**一对多**。

### 4.2 books 与 borrow_records

- 一本图书可以被不同用户多次借阅；
- 一本图书对应多条借阅记录；
- 关系类型：**一对多**。

### 4.3 categories 与 books

- 一个分类下可以有多本图书；
- 一本图书只属于一个分类；
- 关系类型：**一对多**。

## 5. 索引设计

为提高查询性能，可建立以下索引：

### 5.1 users 表

- `username` 唯一索引  

用途：加快登录校验速度，并防止用户名重复。

### 5.2 categories 表

- `name_zh`
- `name_en`

用途：加快分类名称检索。

### 5.3 books 表

- `title_zh`
- `title_en`
- `author`
- `category_id`
- `status`

用途：提升图书搜索、分类筛选、状态筛选效率。

### 5.4 borrow_records 表

- `user_id`
- `book_id`
- `status`
- `borrow_date`

用途：提升借阅记录按用户、图书、状态、时间查询的速度。

## 6. 数据完整性与安全性设计

### 6.1 数据完整性

为保证系统数据正确性，本系统采用以下措施：

1. 主键唯一约束，保证记录唯一性；  
2. 外键约束，保证用户、图书、分类和借阅记录之间的关联正确；  
3. 图书库存控制，防止借出数量超过可借数量；  
4. 删除分类前校验是否仍有关联图书；  
5. 删除图书时校验是否存在借阅记录。  

### 6.2 数据安全性

为保证系统安全性，采用以下措施：

1. 用户密码使用哈希加密存储；  
2. 登录认证采用 JWT Token；  
3. 后端接口根据角色进行权限校验；  
4. 前端通过路由守卫控制访问权限；  
5. 对关键操作增加表单校验和异常处理机制。

## 8. 数据库设计总结

本系统数据库设计围绕图书管理业务展开，结构清晰，关系明确，既能满足管理员对图书、分类、用户和借阅的统一管理需求，又能支持普通读者进行图书浏览与借阅操作。数据库设计具有以下特点：

1. 结构简洁，适合中小型管理系统；  
2. 满足毕业设计项目的完整业务需求；  
3. 支持中英双语字段设计；  
4. 便于与 Flask + SQLAlchemy 集成；  
5. 易于后续扩展，如加入罚金管理、预约管理、操作日志等功能。
